# アーティスト・ライブ情報 収集ルール

> 配置先: `live-ticket-data/COLLECTION_RULES.md`（データ配信リポジトリ側）
> 対象: Claude Code（データ収集エージェント）
> 目的: ライブチケット申込管理アプリ向けに、申込締切情報を正確に収集・配信する。

---

## 0. 大原則

- **「今申し込める」情報が最優先**。チケット申込締切の管理がアプリの目的。
- **正確性 > 網羅性**。曖昧な情報は入れない（空欄/nullにする）。
- **情報源と最終確認日を必ず記録**。ユーザーには「公式確認を促す」前提。
- 個人情報・有料情報は扱わない。公開情報のみ。

---

## 1. 収集対象の選定

### 選定の基本思想：幅と深さを分けて考える

アーティストの登録には2つの役割がある:

| 役割 | 対象 | データ量 |
|------|------|---------|
| **存在感の担保（広く・薄く）** | メジャー・有名アーティスト | アーティスト登録のみ。現役ツアーがあれば付ける程度 |
| **差別化データ（狭く・深く）** | 全通勢が多い中規模アーティスト | ツアー・全公演・全抽選種別まで充実 |

> ユーザーが検索したとき「〇〇がいない」とならないよう、有名アーティストは薄くても登録する。
> 一方でアプリの差別化（申込情報の自動取得）は「1ツアーで何度も申し込む層」に集中させる。

---

### 申込密度（優先度の核心指標）

```
申込密度 = 1ツアーあたりの公演数 × 抽選種別数
```

申込密度が高いアーティスト = マイチケットの価値が最大化するアーティスト。
ツアー・公演・抽選データを**深く入れる優先度はこの指標で決める**。

**高申込密度の典型例**:
- 全国ツアーで公演数が多い（10公演以上）
- FC先行・一般先行・プレイガイド別など抽選種別が複数
- 全通・遠征文化が強いファン層がいる

**低申込密度の典型例**（アーティスト登録はするがデータを深くしなくてよい）:
- スタジアム1〜2公演のビッグアーティスト
- ワンマン年1回程度のアーティスト

---

### 優先順位

1. **ユーザー指定アーティスト**（最優先・深く入れる）
2. **高申込密度アーティスト**（深く入れる）
   - ライブ・全通文化が強いバンド（例: ゴールデンボンバー等のライブバンド）
   - ライブアイドル（ハロプロ系・地下アイドル等・公演数が多い）
   - 全国ツアーを年複数回行う中規模ソロ
3. **メジャー・有名アーティスト**（薄くでも登録する・存在感の担保）
   - 現役ツアー中ならツアーデータも付ける
   - ライブがない時期はアーティスト登録のみでよい
4. **そのうち現在ツアー中／申込受付中**のものは、公演・抽選も併せて収集（鮮度が重要）

---

### ジャンルバランス

均等バランスより申込密度を優先する。

**積極収集ジャンル**（申込密度が高い傾向）:
- バンド（ライブバンド・ビジュアル系・パンク・ロック）
- ライブアイドル（ハロプロ・地下アイドル・ライブに力を入れているグループ）
- 全国ツアーが多い中規模ソロアーティスト

**薄く広くでよいジャンル**:
- K-POP（チケット管理の文化・流通が異なる場合が多い）
- アニメ・2.5次元（イベントが単発・特殊なケースが多い）
- 声優（ライブ頻度が低めの場合が多い）

---

### 規模

- 近期目標: **50組**（メジャー20組程度 + 高申込密度30組程度）
- 最終目標: **200組**（§5.3 のティア別更新体系で運用）
- 追加ペース: 月20〜30組（`/add-artists` で実施）

---

## 2. 情報源ルール

### 優先順位（信頼できる順）
1. **公式サイト / 公式FCサイト**（最優先・一次情報）
2. **公式が明記する正規チケットサイト**（ぴあ / e+ / ローソンチケット / チケットボード 等）
3. 上記が取れない場合は **収集しない**（憶測で埋めない）

### 禁止
- 個人ブログ・まとめ・SNSの非公式情報を一次情報として使わない
- 転売サイトは参照しない
- 有料会員限定でログインが必要な情報は取得しない

---

## 2.5. 【最重要】日程データの取得ルール（違反厳禁）

> アプリの根幹は「申込期限の管理」。誤った日程はユーザーに直接損害を与える。

### 絶対禁止事項
- **WebSearch の AI 生成サマリーから日程を転記してはならない**  
  （検索AIは日程を混同・創作することがある。実害が大きい）
- **異なる種別の日程を別種別に流用してはならない**  
  （例: NF member抽選の日程を e+ 抽選の日程として使用することは禁止）
- **存在を確認していないURLをパスで推測して試すことは禁止**  
  URLは必ず WebFetch で取得したページ内のリンクから辿ること。  
  ページが JS レンダリングでリンクが取れない場合は §2.6 の回避手順に従うこと。

### 必須確認手順
1. `WebFetch` でその種別の公式/正規チケットページを直接取得する
2. 取得したページのテキストに**その種別の日程が明記されている**ことを目視確認する
3. 確認できた日程のみ JSON に入力する。確認できない場合は **null** を入れる

### 根拠引用の義務（完了報告）
- 収集・更新した各抽選（Lottery）の日程について、**取得ページ内の該当記述（日程が書かれている一文）を完了報告に引用**する
  - 例: `entryEndAt=2026-07-20T23:59 ← 「FC先行受付期間：7月10日(金)12:00〜7月20日(月)23:59」（https://...）`
- **引用できない日程は入力してはならない**（= ページに明記されていない → null）
- メインエージェントは commit 前にこの引用と入力値の整合を照合する

---

## 2.6. JSレンダリングサイトの回避手順（追加コストなし）

公式サイトが JS レンダリングで WebFetch で本文が取得できない場合、以下を**順に**試す。
**外部レンダリングサービス・有料ツールは使わない**（コスト制約）。

1. **生HTMLの埋め込みデータを確認**: Bash の `curl -s {URL}` で生HTMLを取得し、以下を探す
   - `<script id="__NEXT_DATA__">` / `window.__NUXT__`（Next.js/Nuxt 系。ニュース・ライブ情報の JSON が丸ごと埋まっていることが多い）
   - `<script type="application/ld+json">`（イベント構造化データ）
   - `<link rel="alternate" type="application/rss+xml" href="...">`（RSSフィードの実在リンク）
   - `<link rel="https://api.w.org/" href="...">`（WordPress の証拠。REST API でニュース一覧が JSON で取れる）
2. **RSSフィードがあれば活用**: WebFetch でフィードを取得して記事URLを辿る。`cache/watch_urls.json` にも登録する（JSサイトは静的HTMLが変化せず更新検知からも漏れるため、RSS登録で検知も回復する。`discover_watch_urls.py` はフィードを自動検出する）
3. **正規チケットサイトに切り替え**: e+ / ぴあ / ローチケ等のアーティストページを一次情報として使う（§2 の正規情報源）
4. **WebSearch で別の公式ページを探す**（URL発見にのみ使う。AIサマリーからの日程転記は禁止のまま）
5. それでも確認できない項目は **null** とし、完了報告に「JSレンダリングで取得不可」と明記する

> **URL推測禁止の例外（限定）**: 手順1で `<link rel="https://api.w.org/" href="...">` が生HTML内に実在する場合のみ、その href 配下の標準エンドポイント `wp/v2/posts` を使用してよい（API ルート宣言からの標準パスであり推測に当たらない）。それ以外のパス組み立ては引き続き禁止。

### null ルール（再確認）
- 不明な日程・確認できない日程は **必ず null**
- `null` の方が誤日程より遥かにマシ（ユーザーが手入力できるが、誤情報は気づかない）

## 3. データ形式

> スキーマは アプリ側 `docs/design/data-model.md` に準拠（system層）。
> ファイル: `data/artists.json`（アーティスト一覧）/ `data/artist/{artistId}.json`（アーティスト別ツアー・公演・抽選）
> ※ `data/tours.json` / `data/performances.json` / `data/lotteries.json` は廃止済み（v8以降は使用しない）

### 配信ファイル構成
```
data/
  manifest.json           # version + files(artists/venues)のhash + artist別hash
  artists.json            # 全アーティスト一覧（id/name/aliases/genre/imageUrl のみ）
  venues.json             # 会場リスト（pref + name の配列、アプリで会場補完に使用）
  artist/
    {artistId}.json       # そのアーティストの tours/performances/lotteries 全部
```

> `venues.json` はアプリの会場入力サジェスト機能に使用する。収集ルールは CLAUDE.md のテンプレートを参照（形式: `[{"pref": "東京都", "name": "東京ドーム"}]`）。アーティスト収集とは別途管理。

### 必須ルール
- 日時は **ISO8601＋JSTオフセット**（例 `2026-08-11T18:30:00+09:00`）
- **不明な値は null**（推測で埋めない）。特に時刻・入金期限。
- `source`: `"system"` 固定
- `sourceUrl`: 取得元URL（必須）
- `lastVerifiedAt`: 収集日（必須）
- `hidden`: boolean（省略時 false）。true にするとアプリの検索・一覧から除外される。フォロー済みユーザーには引き続き表示される。
- **`performance.kind` はアプリ定義値のみ使用**（`"oneman"` / `"fes"` / `"taiban"` のみ。それ以外は使用禁止）  
  ← アプリが認識しない値を入れてもUIに反映されず、意味をなさない  
  ← **分類に迷っても収集は必ずする**。どれか1つに当てはめること（迷ったら `"oneman"` を選ぶ）。kindが不明なことを理由に公演を収集しないことは禁止。

  | 種別 | 使う場面 |
  |------|--------|
  | `"oneman"` | ワンマン・ファンミーティング・単独公演・ゲスト出演・分類迷ったとき |
  | `"fes"` | フェス（複数アーティストが主体のイベントに出演） |
  | `"taiban"` | 対バン（複数アーティストが同列で共演するライブ） |

### 3.6. performanceIds の収集ルール（連携漏れ防止）

`performanceIds` はアプリが「どの公演に申し込めるか」を判断する核心フィールド。**必ずチケットサイト・公式サイトで対象公演を確認してから入力すること。**

#### 3値の意味

| 値 | 意味 | 備考 |
|---|---|---|
| `null` | ツアー全公演が対象（意図的な全公演指定） | FCメンバー向け全公演先行等 |
| `[]` | **禁止** | `validate.py` が ERROR → push 不可 |
| `["id1", ...]` | 指定した公演のみが対象 | 通常はこれを使う |

#### 判定手順
1. 抽選の一次情報（チケットサイト）を WebFetch で取得する
2. 対象公演の一覧を抽出し、ツアー内の全公演 ID と照合する
3. **対象外公演（指定席対象外・席種制限あり等）を除外したIDリストを入力する**
4. 全公演が対象と明記されている場合は `null` を使う
5. 会場未決定・対象が不明な公演は入れない（→後から追記）

#### よくある間違い
| 間違い | 正しい対応 |
|--------|-----------|
| `"performanceIds": []` で登録 | `validate.py` が ERROR。`null`（全公演）か ID列挙に直す |
| ツアー全公演をコピーして貼る | 除外公演（指定席別枠・席種限定等）を必ず取り除く |
| 除外注意書きを見落とす | 公式の「対象外」「※」「注意」部分を必ず読む |
| `sourceUrl` を省略する | lottery には必ずチケットサイトの URL を記録する（`validate.py` が WARNING） |

### ID命名規則
- artist: 英小文字スネーク（例 `golden_bomber`）
- tour: `{artistId}_{slug}_{year}`（例 `golden_bomber_chuyakunensai_2026`）
- performance: `{tourId}_{place}_{mmdd}`（例 `..._niigata_0811`）
- lottery: `{tourId}_{type}`（例 `..._fc_6th`）
- **既存IDと衝突しないこと**（既存JSONを読んでから採番）

### Tour.prices（料金）
- `prices` フィールドを Tours に追加（nullable）
- 形式: `[{"label": "席種名", "amount": 税込金額(整数・円)}]`
- 席種が1つなら1要素、複数あれば複数要素
- 収集時にAIが公式から料金表記を読み取り、自然な配列に変換する
  - 例: 「全席指定8,800円（税込）」→ `[{"label": "全席指定", "amount": 8800}]`
  - 例: 「指定席10,000円 / スタンド8,000円」→ `[{"label": "指定席", "amount": 10000}, {"label": "スタンド", "amount": 8000}]`
- 料金が不明・記載なし → `"prices": null`（推測で入れない）
- 税込金額を優先。税別のみの場合は label に「税別」を明記

### Artist.aliases（略称）
- `aliases` フィールドを Artists に追加（nullable・文字列配列）
- 公式・一般的に広く使われる略称・読み仮名を収録
  - 例: `"aliases": ["ミスチル"]`（Mr.Children）
  - 例: `"aliases": ["クリーピーナッツ"]`（Creepy Nuts）
  - 例: `"aliases": ["ゴールデンボンバー", "金爆"]`（複数ある場合）
- マイナーすぎる通称・非公式スラングは入れない
- 略称が特に無いアーティストは `"aliases": []` または省略

## 3.5. アーティスト名の名寄せ（正規名称解決）

- 入力が**通称・略称・愛称・表記揺れ**でも、**公式の正規名称に解決してから**収集する。
  - 例: 「ミスチル」→ `Mr.Children` / 「ヒゲダン」→ `Official髭男dism`
  - 例: 「サザン」→ `サザンオールスターズ`
- 解決手順:
  1. 入力名でweb検索し、公式サイト/公式SNSで正規表記を確認
  2. 正規名称を `Artist.name` に採用（`id` は英小文字スネークで採番）
  3. 通称と正規名称が大きく異なる場合、完了報告に「入力『◯◯』→正規『△△』として収集」と明記
- **同名の別アーティストに注意**（同名グループ・解散済み等）。公式で本人性を確認できないものは収集しない。
- 解決できない/公式が特定できない場合は収集せず、その旨を報告する。

## 4. 収集手順（1アーティスト）

> **対象範囲**: 登録済み全アーティストが対象。「一部のみ更新」は NG。
> 例: ミスチルだけ更新して他を放置しない。`/refresh-artists` は全アーティストを巡回する。

1. 入力名を正規名称に解決（§3.5）
2. 公式サイトを特定（web検索）
3. 以下の**2ページを必ず両方確認**する:
   - **LIVE / TOUR ページ**: ツアー情報・公演スケジュール
   - **NEWS ページ**: 単発ライブ・ファンミーティング・フェス出演・特別イベントの告知  
     ← ツアーページに掲載されないイベントが多い。必ずNEWSも確認すること（§4.6）  
     ← **一覧ページだけで判断しない**。ライブ・チケット関連の個別記事リンク（`/news/detail/*` 等）を最低1階層 WebFetch で辿ってから「情報なし」「日時不明」と判断すること
4. ツアー名・公演（日時/会場）・抽選（申込締切/発表/入金期限）・料金(prices)を抽出
   - **抽選は全種別を網羅的に取得**（FC先行・一般先行・一次・二次・プレイガイド別など）
   - **受付終了済みの抽選も必ず含める**（ツアー継続中は絶対に省かない。§5.1参照）
   - **会場(venue)は日程とセットで取得する（必須）**: 日程が確定している公演に「会場未定」はほぼあり得ない。アーティスト公式サイトに載っていない場合は**フェス・イベント名で直接検索**して公式サイトから取得すること。検索順: アーティスト公式→フェス公式→チケットサイト（eplus/pia/ローチケ）→SNS公式アカウント。それでも取得できない場合のみ null
5. 取れない項目は null（特に時刻・入金期限・料金は明記なければ null）
6. スキーマに変換（ID採番は既存 `data/artist/{id}.json` を確認）。略称があれば aliases に収録
7. `sourceUrl` と `lastVerifiedAt` を記録
8. 対象アーティストの `data/artist/{artistId}.json` に出力（tours / performances / lotteries を含む1ファイル）
9. `data/artists.json` を更新（アーティストが新規の場合は追加。既存は lastVerifiedAt 等を更新）
10. **セルフチェック（収集漏れ防止）**:
    - [ ] NEWSページで見つけたイベントが全て収集されているか
    - [ ] ツアーが継続中なのに `lotteries: []` になっていないか（全抽選が終了済みでも入れる）
    - [ ] フェス出演が §4.5 のフェスツアーに追加されているか
    - [ ] ファンミーティング等の単発イベントが §4.6 のイベントツアーに追加されているか
    - [ ] **日程が確定している公演に venue: null がないか**（あればフェス公式・チケットサイトで取得してから完了とする）

---

## 4.5. フェス出演の収集

アーティストが出演するフェスも収集対象。フェスはアーティストの公演の1つという位置付け。

### ツアーの作り方

フェス出演はワンマンと同じく Performance だが、tourId が必要なため**フェス用 Tour を1つ作る**。

```json
{
  "id": "{artistId}_fes_{year}",
  "artistId": "{artistId}",
  "title": "{year}年フェス出演",
  "startDate": "（最初のフェス日）",
  "endDate": "（最後のフェス日）",
  "prices": null,
  "source": "system",
  "sourceUrl": "https://（主要フェス公式URL）",
  "lastVerifiedAt": "..."
}
```

- **フェスごとに別のツアーを作る**（年単位まとめは禁止）。ユーザーがどのフェスか識別できないため
- ツアーID: `{artistId}_fes_{fes_slug}` （例: `ado_fes_summersonic_2026`）
- `title`: フェス正式名称（例: `SUMMER SONIC 2026`）。`eventName` と同じ値を入れる
- 同一フェスで複数日程がある場合は1ツアーにまとめてよい（例: 大阪・東京の2公演）
- `endDate` は最終出演日を入れる

### Performance の書き方

```json
{
  "id": "{fesTourId}_{フェス略称}_{mmdd}",
  "tourId": "{artistId}_fes_{year}",
  "venue": "フェス会場名（正式名称）",
  "performanceAt": "出演日T時刻（不明な場合 12:00:00）",
  "kind": "fes",
  "eventName": "ROCK IN JAPAN FESTIVAL 2026"
}
```

- `kind`: **`"fes"`**（`"festival"` は誤り。アプリは `"fes"` のみ認識する）
- `eventName`: フェス正式名称（省略・略称は使わない）
- 出演時刻が未定の場合は `12:00:00+09:00` とし `null` にしない（アプリの日付表示が壊れるため）
- **他の出演者は収集しない**（このアーティストの出演情報のみ）

### 収集ルール

- 出演が公式サイト・フェス公式で「出演決定」と確認できる場合のみ収集（未確定・噂は入れない）
- フェスに申込/抽選がある場合のみ Lottery を作る（フリー入場・当日券のみの場合は不要）
- 通常のツアー収集（§4）と同時に公式サイトのフェス出演情報もチェックすること
- **venue は日程とセットで必ず取得**。アーティストの公式サイトに会場が載っていない場合は**フェス名で検索**してフェス公式サイトから取得する。フェス名が確定している時点で会場も確定しているはずなので、null を入れる前にフェス公式を必ず確認すること

---

## 4.6. ファンミーティング・単発イベントの収集

ワンマンツアーに属さないイベント（ファンミーティング・単発ライブ・生誕ライブ・特別イベント等）も収集対象。NEWSページ（§4 手順3）で発見することが多い。

### ツアーの作り方

年単位でまとめる「イベントツアー」を1つ作る（フェスツアーと同様の考え方。混在させない）。

```json
{
  "id": "{artistId}_event_{year}",
  "artistId": "{artistId}",
  "title": "イベント正式名称（例: M!LK ファンミーティング 2026）",
  "startDate": "（最初のイベント日）",
  "endDate": "（最後のイベント日）",
  "prices": null,
  "source": "system",
  "sourceUrl": "https://（公式サイトURL）",
  "lastVerifiedAt": "..."
}
```

- `title` は**イベント正式名称を使う**（`{year}年イベント` のような汎称は禁止）
- 同じ年に複数の別イベントがある場合は、イベントごとに別のツアーを作る（`{artistId}_event_{year}_02` 等）
- フェスツアー（§4.5 `{artistId}_fes_{year}`）とは別に管理する
- `endDate` は年内の最終イベント日を入れる。新しいイベントが判明したら更新する

### Performance の書き方

```json
{
  "id": "{eventTourId}_{イベント略称}_{mmdd}",
  "tourId": "{artistId}_event_{year}",
  "venue": "会場名（正式名称）",
  "performanceAt": "開演日T時刻（不明な場合は 12:00:00+09:00）",
  "kind": "oneman",
  "eventName": "イベント正式名称（例: M!LK ファンミーティング 2026）"
}
```

- `kind`: `"oneman"`（ファンミーティング・単発ライブ）または `"taiban"`（複数アーティスト出演の対バン形式）を使う
- `eventName`: イベント名を必ず入れる（通常ツアー公演との区別のため）
- 出演時刻が未定の場合は `12:00:00+09:00`（`null` にしない。アプリ表示が壊れるため）

### 収集ルール

- 公式サイト・公式SNSで「開催決定」と確認できる場合のみ収集（未確定・噂は入れない）
- チケット/申込がある場合のみ Lottery を作る（無料・招待制・イベント参加型は不要）
- フェス出演（§4.5）と明確に区別する（アーティスト主催またはゲスト出演のイベントが対象）

---

## 5. マージ・追記ルール

- 既存の `data/artist/{artistId}.json` を**読み込んでから更新**（全消ししない）
- 同一IDが既にあれば**更新**、なければ**追加**
- 一括収集時も各アーティストのファイルに追記していく
- 最後に **`data/manifest.json` の `version` を +1** し、以下を更新:
  - `files.artists.hash`（artists.json を更新した場合）
  - `artists.{artistId}.hash`（更新したアーティスト分）
- `data/artists.json` の該当アーティストも必ず同期更新する（lastVerifiedAt 等）

### 5.1. 終了データの除外（肥大化防止）

- 終了データの扱いは**ツアー単位**で判断する（個別公演・個別抽選は終了しても即除外しない）。
- 除外の基準:
  - **ツアー（Tour）**: 配下の**全公演が終了**し、かつ紐づく**全抽選の入金期限も全て過ぎた**ツアーは、配下の公演・抽選ごと配信から除外する
  - **個別公演（Performance）**: ツアー内に未来の公演が1つでもあれば、**終了済み公演も残す**（アプリ側がアーティスト詳細で終了公演も表示するため）
  - **個別抽選（Lottery）**: 受付終了済みの抽選でも**除外しない**。ユーザーが申込状況（当選/落選/入金済み等）をステータス管理できる必要があるため。ツアー単位除外のタイミングで一緒に消す。
  - 上記で参照されなくなったアーティストも、他に有効なツアーが無ければ除外可
- **アプリ側はユーザーが申込状況を付けた時点でスナップショット保持**するため、配信から消してもユーザーの記録は失われない（削除はユーザーの手動のみ）。
- 迷う場合は「未来の公演が1つでもあるTourやArtistは残す」。

### 5.1.1. artist/{id}.json の肥大化対策

- v8以降はアーティスト単位ファイル（`data/artist/{id}.json`）で管理するため、単一ファイルの肥大化は解消済み。
- 各ファイルの肥大化抑制:
  - **ツアー単位除外（§5.1）を確実に実行**: 完全終了したツアーは公演ごと消えるため、無限には増えない
  - **`/refresh-artists` 実行時に必ず終了ツアーの掃除を行う**（鮮度更新と除外をセットで）
  - 1アーティストが極めて多数のツアー・公演を持つ場合は年度単位分割を検討（その時点で設計タスク化する）

### 5.2. 登録済みの再収集（鮮度更新）

- 抽選は世代交代する（例: 5次→6次受付）。**登録済みアーティストも定期的に再収集**して最新化する。
- 再収集は新規追加ではなく**既存IDの更新**（§5の同一ID更新ルールに従う）。
- 新しい抽選（次受付）が出ていれば Lottery を追加/更新、終了した抽選は §5.1 で除外。
- → コマンドは §5.3 のティア別更新体系を参照。

---

### 5.3. スケール対応：ティア別更新サイクル（200組対応）

100組を超えると全アーティストの週次更新は非現実的になる。アクティビティに基づきティアを自動判定し、更新頻度を分ける。

#### ティア定義（`artist/{id}.json` の内容から動的判定・手動タグ不要）

| ティア | 判定条件 | 更新頻度 |
|---|---|---|
| **Hot** | `lotteries` に `entryEndAt` が現在から3ヶ月以内（未来）のものがある | 毎週 |
| **Cold** | 上記に該当しない（ツアー予定あり・なし問わず） | 月次 |

#### コマンド体系

| コマンド | 対象 | 推奨タイミング |
|---|---|---|
| `/refresh-smart` | 公式サイトに変化があったアーティストのみ（自動判定） | 毎週（推奨） |
| `/refresh-hot` | Hot tier のみ（動的判定→絞り込み→鮮度更新） | 毎週（smartが使えない場合） |
| `/refresh-all` | 全アーティスト + 終了ツアー掃除（§5.1） | 月次 |
| `/add-artists` | 新規アーティストの追加バッチ | 月次 |

#### 実行の原則（競合回避・リミット対策）

**1組×5並列、5組完了ごとにcommit** の方式を使う。

- **バッチサブエージェント**: 1組を1エージェントが担当。`data/artist/{担当id}.json` のみ書く。`artists.json` / `manifest.json` には触らない
- **メインエージェント**: 5組完了後に `artists.json` を一括更新 → `validate.py` → commit & push。全バッチ完了後に `update_manifest.py` → commit & push
- 5組完了ごとに push することで、途中でリミットに達しても完了分は確定する

```
グループ①: 5つのサブ（各1組）を同時起動（background）
  ↓ 全5つの完了を待つ
メイン: artists.json を5組分一括更新 → validate.py → commit & push
  ↓
グループ②: 次の5組で繰り返し
  ↓ 全グループ完了
メイン: update_manifest.py → commit & push
```

#### /refresh-smart の手順

**差分チェックで変化があったアーティストだけを更新する。セッション消費を最小化できる。**

```
1. python3 tools/check_updates.py 2>/dev/null > /tmp/changed.txt
   → 変化ありアーティストIDが1行ずつ出力される
   → 初回（cache なし）は全件が出力される（期待通り）

2. cat /tmp/changed.txt で対象IDを確認する

3. 対象IDを5組ずつのグループに分割

4. グループごとに1組×5並列で起動
   - 各サブ: §4 の収集手順を実施（終了ツアー掃除は不要、変化があった分のみ）
   - 全完了後: artists.json の lastVerifiedAt を更新 → validate.py → commit & push

5. 全グループ完了後:
   python3 tools/update_manifest.py → commit & push
   git add cache/source_hashes.json && git commit -m "update: source hash cache" && git push origin main
```

**注意事項:**
- `cache/source_hashes.json` は git 管理する（次回チェックの基準点）
- 公式サイトを更新していないアーティストは出力されないため、スキップして正しい
- 公式サイトが存在しない（sourceUrl が null）アーティストは常に"変化あり"扱いとなる

**監視URLの複数化（見落とし対策）:**
- `cache/watch_urls.json`（git 管理）にアーティストごとの追加監視URL（NEWS / LIVE ページ等）を登録すると、`check_updates.py` が公式TOPと合わせて監視する
- 登録は `python3 tools/discover_watch_urls.py {artistId}` で公式TOPページの実在リンクから自動抽出する（URLのパス推測は禁止。手動追記も可）
- **新規アーティスト追加時は必ず discover_watch_urls.py を実行**して監視URLを登録すること

#### /refresh-hot の手順
1. 全 `data/artist/{id}.json` を読み込み、Hot tier アーティストを抽出
   - 判定: `lotteries[].entryEndAt` が今日から90日以内かつ未来のものが1件以上あるか
2. 5組ずつのグループに分割し、グループごとに1組×5並列で起動
3. グループごとに全完了後、メインが `artists.json` の `lastVerifiedAt` を更新 → validate → commit & push
4. 全グループ完了後: `python3 tools/update_manifest.py` → commit & push

#### /refresh-all の手順
1. 全アーティストを5組ずつのグループに分割し、グループごとに1組×5並列で起動
   - 各サブ: §4 の収集手順 + 終了ツアーの掃除（§5.1）を実施
2. グループごとに全完了後、メインが `artists.json` の `lastVerifiedAt` を更新 → validate → commit & push
3. 全グループ完了後: `python3 tools/update_manifest.py` → commit & push

#### アーティスト追加ペース（目安）
- 月20〜30組バッチで `/add-artists` を実行
- 50 → 200組 = 約5〜7ヶ月

## 7. 品質チェック（push前）

**必ず以下を実行すること：**

```bash
python3 tools/validate.py
```

✅ が表示されたら push OK。エラーがあれば修正して再実行。

---

### validate.py がチェックする項目（参考）
- 全 JSON ファイルがパース可能
- artists.json の必須フィールド（id/name/source/sourceUrl/lastVerifiedAt）
- artists.json の ID と artist/ ディレクトリのファイルが一致
- Tour / Performance / Lottery の必須フィールド
- ID参照整合性（performance.tourId が存在するか等）
- 日時フォーマット（ISO8601）
- グローバルID一意性
- manifest hash 整合性（不一致は WARNING・push 前に update_manifest.py を実行）
- **意味的整合性（ERROR）**: 日時の前後関係（受付開始＞締切、当落発表＜締切、入金期限＜当落発表、開場＞開演 等）
- **カバレッジ（WARNING）**: 未来公演があるのに抽選ゼロのツアー / 未来公演の venue が null / 料金の異常値 等
  ← **WARNING は「抜け漏れの疑い」**。push 前に必ず内容を確認し、収集漏れなら追加収集してから push する

---

### validate.py がチェックできない項目（目視確認）
- [ ] 推測で埋めた値がない（不明は null）
- [ ] sourceUrl が有効な公式URL
- [ ] lastVerifiedAt が今日の日付

---

## 8. 注意事項

- 情報は参考情報。アプリ側で「必ず公式確認を」と案内する前提。
- 古くなった情報はアプリ側 sync で `stale` 化されるので、**削除より更新を優先**。
- 迷ったら入れない（誤情報はアプリの信頼を損なう）。
